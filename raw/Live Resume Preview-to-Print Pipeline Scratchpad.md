

This note describes the active styled-resume pipeline in the live local tree. It is written as an operator / LLM scratchpad, not as product documentation.

Use this note when you need to understand what is implied between the live preview, the Export Styled PDF click, the parser/export request, the worker bootstrap, /print/resume, and the final PDF bytes generated by page.pdf().

## Active-code authority

- Treat my-app/ as the active implementation surface for styled resume preview and styled PDF export.
- Treat my-app/src/lib/layout/documentAppearance.ts as the live document appearance source.
- Treat backup trees, archive folders, pdf-ingest/, and other legacy parser/export paths as non-authoritative unless they are explicitly being used as historical context only.
- The styled-resume parity path should be reasoned from the active code below, not from older snapshots or assumptions about previous renderer behavior.

## Pipeline overview

The styled resume export path is one continuous flow:

1. CvForge holds the active CV document, the selected style preset, and the current live preview surface.
2. VerbatiResumePreview resolves the style preset into CSS/theme vars and renders ResumePage.
3. ResumeExportControl triggers exportDocumentFile for kind: "resume", format: "pdf", mode: "styled".
4. exportDocumentFile posts the request body to the parser/export endpoint at /api/v1/document-export/resume/pdf.
5. The parser/export service invokes document-export-worker.ts.
6. For styled preview-driven exports, the worker builds a ResumePrintRoutePayload, launches Playwright, and opens /print/resume.
7. ResumePrintPage reads the injected payload from window.__DASTI_RESUME_PRINT_PAYLOAD__, rebuilds the same theme vars, renders ResumePage, waits for document.fonts.ready, and reports a ready snapshot.
8. The worker captures the pre-page.pdf() surface, then calls page.pdf() and returns the exact PDF bytes to the browser.

The core implication is that styled preview and styled PDF are supposed to be twins because both surfaces render through the same ResumePage component using the same style preset and renderer variant.

## Live preview path

### UI entrypoints

- my-app/src/pages/CvForge.tsx
- my-app/src/features/verbati/VerbatiResumePreview.tsx
- my-app/src/features/verbati/resume/ResumePage.tsx

### Style resolution

The live preview path resolves styling in this order:

1. CvForge holds the selected VerbatiStylePreset.
2. VerbatiResumePreview calls buildVerbatiThemeVars(stylePreset).
3. resolveVerbatiStyle normalizes the preset.
4. VERBATI_LAYOUT_TO_RENDERER resolves the rendererVariantId.
5. ResumePage renders the selected layout variant with those CSS vars on the preview surface.

This means the preview is not just “using some CSS”. It is using:

- a normalized stylePreset
- a resolved rendererVariantId
- live theme vars from buildVerbatiThemeVars
- the active ResumePage layout variant

If preview typography looks wrong, the first questions are:

- did the selected preset normalize the way you think it did?
- did the right rendererVariantId get selected?
- did the CSS vars land on the preview wrapper?
- did the font families only resolve as strings, or did the font faces actually materialize?

## Export click path

### UI action

The export action comes from:

- my-app/src/components/ResumeExportControl.tsx
- my-app/src/pages/CvForge.tsx
- my-app/src/lib/exportDocumentFile.ts

When the user clicks Export Styled PDF, CvForge calls exportDocumentFile(...) with:

- kind: "resume"
- format: "pdf"
- mode: "styled"
- data: a ResumePreviewPrintSource
- stylePreset
- fileNameBase
- optional debug metadata such as ResumeTypographyAuditMetadata

### Request contract

The live contracts are defined in my-app/src/lib/document-export-models.ts.

Important types:

- ExportDocumentFileArgs
- ResumePreviewPrintSource
- ResumePrintRoutePayload
- ResumeTypographyAuditMetadata

ResumePreviewPrintSource is the critical styled-resume export shape. It carries:

- kind: "resume"
- renderSource: "preview"
- resumeData
- stylePreset
- rendererVariantId

That is the handoff from preview land into export land. If parity fails, this is the first transport boundary to inspect.

## Parser/export worker path

### Network boundary

exportDocumentFile serializes the request body and posts it to:

- /api/v1/document-export/resume/pdf

The client does not render the PDF itself. It requests the export service, receives bytes back, then triggers the browser download.

### Worker boundary

The styled export worker path lives in:

- my-app/scripts/document-export-worker.ts

For styled resume preview exports:

1. The worker receives a payload whose data is a ResumePreviewPrintSource.
2. It calls buildResumePrintRoutePayload({ data }).
3. It writes or exposes debug data such as the bootstrap payload and print snapshot when audit metadata is present.
4. It launches Playwright.
5. It injects the payload into the page via window.__DASTI_RESUME_PRINT_PAYLOAD__.
6. It navigates to /print/resume.
7. It waits until the print route reports a ready state.
8. It captures a pre-page.pdf() screenshot.
9. It calls page.pdf(...).

The worker is not supposed to invent a second style system. Its job is to rehydrate the same preview-intent payload into the print route.

## Print route path

### Route

The active print route is:

- /print/resume

It is wired in:

- my-app/src/App.tsx
- my-app/src/pages/ResumePrintPage.tsx

### Print page responsibilities

ResumePrintPage does the following:

1. Reads window.__DASTI_RESUME_PRINT_PAYLOAD__.
2. Validates it as a ResumePrintRoutePayload.
3. Rebuilds the same theme vars with buildVerbatiThemeVars(payload.stylePreset).
4. Renders ResumePage using:
    - payload.resumeData
    - payload.rendererVariantId
    - payload.stylePreset
5. Waits for document.fonts.ready.
6. Waits for animation frames so the surface settles.
7. Publishes ready/debug state through window.__DASTI_RESUME_PRINT_STATUS__.

Important browser globals on the print route:

- window.__DASTI_RESUME_PRINT_PAYLOAD__
- window.__DASTI_RESUME_PRINT_BOOTSTRAP__
- window.__DASTI_RESUME_PRINT_STATUS__

The print route is meant to be a twin rendering surface, not a special export-only renderer.

## Final PDF generation path

Once /print/resume reports ready, the worker captures the same rendered surface that the print route is showing and then calls:

- page.pdf({ format: "A4", printBackground: true, preferCSSPageSize: true, ... })

The returned PDF bytes come from Playwright’s print pipeline, not from a separate HTML-to-PDF template for this styled path.

That means the styled export parity chain is:

- same resumeData
- same stylePreset
- same rendererVariantId
- same ResumePage
- same theme vars
- same font materialization
- same print-ready rendered surface

If the PDF differs after all of that, the divergence is downstream of the print route, usually in print timing, font readiness, or PDF generation itself.

## Same-export parity contract

For a single styled export run, preview and styled PDF are supposed to remain twins across these invariants:

- same selected stylePreset
- same rendererVariantId
- same resumeData
- same CSS/theme vars from buildVerbatiThemeVars
- same resolved heading/body font families
- same materialized font faces actually loaded in the browser
- same ResumePage layout variant
- no alternate export-only style system
- no fallback renderer for styled resume export

If any of those stop matching, parity is already broken before the PDF bytes are produced.

## Why typography can look wrong

Typography mismatches can come from different boundaries. The common cases are:

### 1. Computed font-family strings match, but the real font never materializes

This is the classic “CSS says Syne, but the page is not actually using Syne” failure.

What to inspect:

- local font-face injection
- document.fonts
- whether the font family is actually loaded on preview and print
- whether the PDF declares or visibly uses the expected family

### 2. The font family is correct, but the role styling is intentionally different

A block can use the body font family but still look nothing like normal body copy because the layout gives it:

- uppercase
- heavier weight
- tighter line-height
- different letter-spacing
- display-scale sizing

This is especially relevant for summaries and hero text in some layout variants. A “wrong-looking summary” is not always a wrong font-family.

### 3. Preview and print use different wrapper surfaces

Even with the same ResumePage, wrapper differences can change the apparent result:

- different width constraints
- different scale/zoom
- different print-route shell styles
- different inherited base font or CSS vars

That is why the audit chain compares preview screenshot vs pre-page.pdf() print screenshot, not just payloads.

### 4. Worker or PDF timing issue

If /print/resume is correct but the exported PDF is wrong, the likely boundaries are:

- fonts not actually ready before page.pdf()
- page not settled yet
- print media differences
- Playwright PDF generation timing

This is why the worker-side pre-page.pdf() screenshot is mandatory in the parity harness.

## Debug checkpoints

When styled preview and styled PDF do not match, inspect the same export run in this order:

1. selected preset in the UI
2. preview snapshot and computed heading/body fonts
3. export request body from the real Export Styled PDF click
4. worker bootstrap payload
5. /print/resume snapshot immediately before page.pdf()
6. pre-page.pdf() screenshot
7. exact returned PDF bytes
8. rasterized PDF page comparison

Do not skip directly from “preview looks wrong” to “PDF bytes differ”.

The proven parity harness path in this repo already follows that chain and saves these artifacts:

- ui-export-request.json
- preview.json
- worker-bootstrap.json
- print-route.json
- print-route-pre-pdf.png
- returned-export.pdf
- pdf-raster-page-1.png
- image diffs and summary JSON
